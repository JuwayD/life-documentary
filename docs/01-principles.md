# 开发方针

## 核心理念:代码不做决策

这是贯穿整个项目的**第一原则**:

```
代码 = 手脚 (读写数据、改状态、提供工具)
AI  = 大脑 (决策、判断、创造)
```

代码从不回答"应该怎样",只回答"要我读什么/写什么/算什么"。AI 决定一切,但 AI 只能通过工具间接操作世界。

## 子方针

### 1. AI 友好 (AI-Friendly)

本项目的工具链、接口、工作流均以"AI 能高效协作"为设计目标。

- **工具签名正交** — 每个工具做一件事、做好、只做一件事
- **入参出参结构化** — 所有工具返回 JSON,documentation 内嵌在 schema 中
- **错误信息可操作** — 错误消息包含修复建议,">- 例:`{error: 'entity not found', suggestion: 'use list_entities_nearby to find valid ids'}`
- **一规则一文件** — 代码按功能拆分,AI 修改一个文件时上下文窗口刚好
- **测试名即文档** — 测试用例的命名就是需求清单:`test_平民攻击知府_法条被引用`

### 2. 人类可审核 (Human-Auditable)

AI 的每一个决策都要能被人类追踪、理解、重现。

- **全量决策日志** — 每轮交互记录:玩家输入 → AI 思考 → 工具调用 → 世界变更 → 输出叙述
- **决策引用溯源** — AI 引用法律/民俗时必须携带来源 ID,人类可直接查阅原文
- **回放模式** — 给定世界快照和玩家输入,可重现 AI 的决策链条
- **Dry-Run 模式** — 所有写操作只打印不执行,人类审阅后可决定是否生效

### 3. TDD (测试驱动开发)

**代码先过测试,才能进仓库。**

- **红-绿-重构** 循环:先写失败的测试,再写最小实现通过它,最后重构
- **工具层**:传统 TDD,测试工具读写逻辑的正确性
- **AI 层**:契约测试 + 场景回放,测试 AI 决策的约束而非内容
- **集成层**:给定固定场景,检查 AI 调用工具的模式是否符合预期

### 4. 渐进交付

- **Phase 0**:纯文字闭环 — 能跑通"输入 → 决策 → 读写 → 输出"的最小循环
- **Phase 1+**:每次迭代增加可玩性,不追求一步到位
- **至少垂直,不要水平**:每个 Phase 都跑通端到端,再扩展

### 5. 开发过程即产出物

本项目的开发过程本身是有价值的:

- **ADR**:每次重大决策写架构决策记录,说明背景、方案、取舍
- **会话日志**:每次开发会话结束自动生成摘要,记录"为什么这么做"而不仅是"做了什么"
- **复盘记录**:遇到困难、踩坑、转向时专门记录,供后续分享
- **每条 ADR 有日期和状态**:proposed → accepted → deprecated,让读者能看到决策的生命周期
