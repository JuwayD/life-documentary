# ADR-0099: 复盘分页与结局进度追踪

**状态**: 已接受
**日期**: 2026-06-11

## 背景

Phase 8 信息展示优化的最后一项路线图任务：「继续优化信息展示（结局进度、复盘回合上限等）」。

两个具体问题：
1. **复盘回合上限**：前端硬编码 `limit=50`，服务端上限 100，长局玩家无法回顾早期回合。
2. **结局进度**：结局面板只展示已完成结局或种子列表（上限 3 个），无进度指示器。

## 决策

### 复盘分页

- 服务端 `/api/audit` 上限从 100 提升至 500，新增 `offset` 参数，返回 `total` / `has_more` 字段。
- 前端复盘弹层增加「加载更早回合」按钮，每次加载 50 回合，已加载回合数/总数实时显示。
- 旧回合插入时间线顶部（保持时间顺序），筛选/搜索功能跨全部已加载回合生效。

### 结局进度追踪

- 新增 `update_ending_progress` 工具：GM 为每个结局方向注册步骤（step_id + step_label + completed），代码只存储状态，不做条件判断（延续 ADR-0017 设计原则）。
- 前端结局面板为每个结局种子显示进度条 + 步骤清单（✓/○），移除 3 个种子的显示上限。
- 步骤数据存储在 `ending_steps` flag 中，按 `ending_id` 分组。

## 不采用的方案

- 虚拟滚动/无限滚动：回合数上限 500，DOM 节点可控，分页按钮足够。
- 代码层定义结局条件：违反项目核心原则（AI 决策，代码只读写数据）。

## 影响

- 新增 1 个写工具（`update_ending_progress`），1 个读工具变更（`list_endings` 返回 `ending_steps`）
- 前端 review modal 和 ending panel 交互变更
- 12 个新单元测试
