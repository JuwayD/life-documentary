# ADR-0008: Phase 4 渲染层与流式叙述

- **状态**: Accepted
- **日期**: 2026-06-08
- **关联**: docs/04-roadmap.md (Phase 4)

## 背景

Phase 0–3 已交付:三层架构、Web UI、3 场景 12 地点 13 实体、时辰/金钱/法条、回合制战斗与 D20 检定。

按 roadmap,Phase 4 目标是"加上视觉"。当前 Web UI 是文字面板(index.html + style.css + app.js),没有任何 2D 场景渲染。本 ADR 决定 Phase 4 的边界与技术选型,**目标是补一层视觉,而不是重写游戏**。

## 决策

### 1. 渲染库选 PixiJS,不选 Phaser

- **PixiJS**: 纯渲染库,WebGL 加速 sprite/容器,~400KB。我们不需要 Phaser 自带的物理引擎、Tween、状态机 — 那些会和 ADR-0001("代码是手脚,AI 是大脑")冲突。
- **不选 canvas 手写**: 大量等距瓦片排序、sprite 批渲染、相机平移自己写得不偿失。
- **不选 three.js**: 2.5D 等距瓦片不需要真 3D,GPU 内存和包体都过剩。

### 2. 渲染层是"State Layer 的镜像",不是状态源

PixiJS 场景图由前端从 `/api/snapshot` 拉到的 World 快照**派生**:

```
SQLite (源) → /api/snapshot (REST) → JS world model → PixiJS scene graph
```

- 玩家位置变化 → 后端 move_entity → 下一次 snapshot → 前端 diff → sprite 重新定位
- 前端不维护任何"游戏状态",只维护"渲染状态"(相机、动画进度、UI 焦点)
- 这保证了 CLI/Web 两套入口看到的世界完全一致

### 3. 等距瓦片格式

- 每个 location 对应一张瓦片地图,瓦片尺寸 64×32(标准等距 2:1)
- Location 的 `size: [w, h]` 字段就是瓦片网格尺寸
- Entity 的 `pos: [x, y]` 直接映射到瓦片坐标
- 第一版只用 4 张占位瓦片:石板地、木地板、墙、门 — 美术后补

### 4. 角色精灵

- 每个 entity_id 对应一个 `static/sprites/{id}.png`,缺省走 `_default_npc.png`
- 4 方向 × 2 帧待机动画(共 8 帧),先用站立单帧占位
- 战斗中受击时整精灵闪红 200ms(纯前端动画,不依赖后端状态)

### 5. 流式叙述用 WebSocket

REST 模式下,GM Agent 一次 turn 可能耗时 3-8 秒,玩家干等没有任何反馈。新增 `/ws/turn` 端点:

- 玩家输入 → WS 推送 → 后端逐 token 转发 Claude 流式响应
- 工具调用进度也走 WS(e.g. `{"type":"tool_call","name":"query_laws"}`)
- 前端边收边渲染叙述,工具调用在审计面板实时滚动

REST 端点 `/api/turn` 保留,作为 CLI/E2E 测试的同步入口。

### 6. UI 布局重构

```
┌──────────────────────────────────────┬──────────────┐
│                                      │   状态面板    │
│         PixiJS 视口(场景)            │   (玩家)     │
│         相机跟随玩家                  │   (NPC 列表) │
│                                      │   (时辰金钱)  │
│                                      │              │
├──────────────────────────────────────┤              │
│   叙述输出(流式渲染,可滚动历史)        │   审计面板    │
├──────────────────────────────────────┤   (工具调用)  │
│   玩家输入框                          │   (引用法条)  │
└──────────────────────────────────────┴──────────────┘
```

明代纸质风格保留(右侧面板),左侧场景视口是新增的视觉区。

### 7. 战斗反馈最小化

- 受击闪红
- HP 变化时实体头顶弹一个 `-5` 飘字,500ms 上浮淡出
- 不画格子、不画攻击距离、不画轨迹(那会让 LLM 临场决策变得"被画面绑架")

## 不引入

- ❌ 物理引擎(碰撞、重力)
- ❌ 前端寻路 A*(移动由 LLM 决定,不是玩家自由点击)
- ❌ 鼠标点击地块移动(保持"输入文字 → AI 决策")
- ❌ 战斗 UI 特殊面板(攻击按钮、技能栏)
- ❌ NPC 自动行走动画(NPC 只在被交互时才"活",见 02-architecture)
- ❌ 美术资源生产(本阶段全用占位图)
- ❌ 音效

## 任务拆分(实施时)

1. **WebSocket 流式端点** — 后端 `web/server.py` + `llm/agent.py` 流式适配
2. **前端 World 镜像** — `static/world_model.js`,负责 snapshot diff
3. **PixiJS 集成** — `static/render/scene.js`,等距瓦片 + sprite
4. **占位美术** — `static/sprites/`,4 瓦片 + 8 角色精灵
5. **UI 重构** — `index.html` + `style.css` 加视口区
6. **流式叙述渲染** — `app.js` 接 WS,逐 token 写入叙述区
7. **战斗反馈** — 受击闪红 + 伤害飘字
8. **E2E 测试** — Playwright 验证视口出现 sprite、流式叙述能滚出来

## 后果

### 正面
- ✓ 视觉反馈大幅改善玩家沉浸感
- ✓ 流式叙述消除"AI 思考期间盯白屏"问题
- ✓ 前后端通过 snapshot 解耦,渲染层崩了游戏逻辑不受影响
- ✓ 选 PixiJS 不引入物理/状态机,继续守住 ADR-0001

### 负面
- ✗ 包体增加 ~400KB (PixiJS)
  - **对策**: CDN 引入,首屏只加载启动场景的瓦片
- ✗ WebSocket 增加部署复杂度
  - **对策**: FastAPI 原生支持,uvicorn 一行配置;REST 端点保留兜底
- ✗ 占位美术不够好看
  - **对策**: Phase 4 的 KPI 是"能动起来",美术留到后续迭代
- ✗ E2E 测试需要等待 PixiJS 初始化
  - **对策**: 前端暴露 `window.__rendererReady` 信号,Playwright 等这个事件
