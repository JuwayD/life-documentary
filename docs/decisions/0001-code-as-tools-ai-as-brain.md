# ADR-0001: 代码作工具,AI 作大脑

- **状态**: Accepted
- **日期**: 2026-06-06

## 背景

设计 AI 驱动的 RPG 时,有三种范式可选:

1. **传统范式**: 代码硬编码所有规则,AI 仅负责自然语言解析和文本生成
2. **混合范式**: 代码做"物理"判定,LLM 做"社会"判定(此前的方案)
3. **AI 原生范式**: 代码只提供工具,AI 是唯一决策者

经过和用户多轮讨论,确认采用范式 3。

## 决策

**代码的全部职责是提供工具 (read / write / util),AI 是唯一的决策者。**

具体表现为:
- 代码层不存在"游戏规则"代码
- 代码不判断行为是否合理、合法、可行
- 代码不计算"该怎么处理",只回答"读什么/写什么/算什么"
- 所有判定、推理、叙述均由 GM Agent (Claude) 完成

## 驱动力

用户原话:**"定死的规则体现不出 AI 游戏魅力"**。

具体场景: 玩家说"我用筷子蘸酒在桌上画八卦请知府看",硬编码规则只会说"未知动作",而 AI 判官能识别这是隐晦暗示。

## 备选方案对比

| 范式 | 灵活性 | 可控性 | 实现成本 | 选择理由 |
|---|---|---|---|---|
| 全硬编码 | ❌ 极低 | ✓ 极高 | 中 | 失去 AI 游戏的魅力 |
| 混合(Judge Agent) | ⚠️ 中 | ⚠️ 中 | 高 | 边界模糊,代码仍越界做决策 |
| **工具范式 (本)** | ✓ 高 | ⚠️ 靠日志+约束 | 中低 | 边界清晰,AI 优势最大化 |

## 后果

### 正面
- ✓ AI 能处理无限多样的玩家创意
- ✓ 代码极简,工具是正交的、易测试的
- ✓ 加新功能 = 加新工具,不需要改决策代码
- ✓ AI 友好程度最高

### 负面
- ✗ AI 可能做出"违反常识"的判定(如把死人复活)
  - **对策**: 强约束系统提示词、强审计、可回滚
- ✗ 每轮 LLM 成本和延迟较高
  - **对策**: 限制每轮工具调用数、缓存常见场景
- ✗ 无法保证 100% 确定性 (同样输入可能不同结果)
  - **对策**: 接受这是 AI 游戏的本质,审计可追溯

## 红线

为防止 AI 范式滑坡,以下情形**必须**走代码硬路径:
- 玩家存档读写
- 真随机数生成 (掷骰)
- 数据持久化
- 数学计算 (距离/坐标)

这些"工具内部实现"是硬编码的,但工具暴露给 AI 的接口仍是"做这件事",而非"决定这件事"。
